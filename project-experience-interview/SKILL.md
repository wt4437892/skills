# Project Experience Interview

## 概览

先做轻量仓库初始化，自行确认项目定位、模块边界和高价值链路，再逐轮推进项目经历问答。输出始终模拟真实项目经历面试现场：每轮只给一道面试官提问和一段能过面试官评估的回答。始终把内容分成三类：`已确认事实`、`合理推断`、`回答边界`，避免把通用八股硬套成项目经历。

## 核心交互规则

- 在初始化完成前，优先限制仓库探索范围，不直接深翻大批代码。
- 默认先依赖仓库事实、配置和代码证据建立上下文，不把用户输入当成唯一来源。
- 如果当前会话已经执行过 `/init`，或已经存在稳定的项目上下文，就直接复用该初始化结果，不要重复做完整摸底。
- 初始化和补证都遵守“最小读取、命中即停”原则：一旦已经足够支撑当前题目，就停止继续读文件。
- 连续回合优先复用上一轮已经确认过的模块、链路和证据，不为每一题重新扫同一批文件。
- 发现信息不足时，先下调断言强度并收窄结论范围，用保守表述继续输出，不要立刻把会话改成信息采集。

## 维度定义

本 skill 只保留 `技术岗位/级别` 这个调整维度，输出形态固定为真实面试中的一问一答。

### 技术岗位/级别

根据仓库主技术栈和模块形态推断，未指定时自动推断：

- `Java 后端`：适用于 Spring/Java/RPC/事务/缓存/SQL/排障等典型后端仓库，重心放在服务拆分、调用链、事务一致性、缓存与数据库协同、线上问题定位。
- `架构师/高级工程师`：适用于跨域模块较多、存在平台化治理、演进路径、容量评估议题的仓库，重心放在边界划分、系统演进、治理策略、容量与稳定性权衡。

## 工作流

### 0. 初始化项目上下文

先读 `references/initialization-playbook.md`，再判断是否可以使用 `references/project-context.md`。`project-context.md` 是当前项目的种子文件，**使用前需替换为实际项目内容**。如果文件头部标注为模板骨架或仍保留其他项目的示例内容，则忽略其中的项目特定结论，改为完全依赖当前仓库入口重建项目卡片。

如果当前会话已经通过 `/init` 建立了项目概览，或前文已经确认过项目定位、核心模块和高价值链路，则把这些内容直接视为初始化基线：

- 不再重复读取初始化入口文件
- 不再重复回建项目卡片
- 直接进入第 1 节定点验证，或在信息已足够时直接进入第 6 节输出第一题

初始化阶段只允许读取这些高价值入口：

- `README.md`
- `pom.xml`
- `package.json`
- `AGENTS.md`
- 仓库根目录模块列表
- `references/project-context.md`
- `references/initialization-playbook.md`

这些入口不是必须全部读取。按信息增益和文件存在性从高到低选择，满足初始化最低条件后立即停止，不补齐“清单式通读”。

初始化时优先确认：

- 项目一句话定位
- 仓库中证据最强的模块、链路或专题
- 最适合转成面试问答的亮点、难点或结果
- 默认岗位推断，具体映射见“岗位映射”章节

初始化未完成时，不继续深挖代码，先补最关键的仓库证据入口；默认不向用户追问职责、岗位或本轮产出偏好。

初始化完成的最低条件：

- 已知项目定位
- 已知至少一个可落到代码证据的模块、链路或主题
- 已形成可持续补全的项目卡片

完成初始化后，转入第 6 节规定的输出流程。

### 1. 定点建立项目画像
初始化完成后，再围绕当前问题做定点验证。只读与当前问题直接相关的模块目录、配置文件、核心 Service、Controller、Mapper，不做全仓库扫描。

定点验证时，先用模块名、接口名、配置项、消息主题、表名等关键词做定位，再决定是否打开具体文件；默认先定位，再阅读，不直接展开目录通读。

优先验证这些与当前问题直接相关的信息：

- 当前模块或链路的关键实现位置
- 当前题目对应的边界、异常流、状态流转和依赖关系
- 当前链路内可落到个人动作的难点、优化点和取舍点
- 支撑当前回答所需的配置项、类名、方法名、表名或消息主题

单轮补证优先控制在 `1~3` 个核心文件或配置点以内；如果这批证据已经足够支撑题目和答案，就不要继续横向扩读同模块其他文件。

把仓库里的明确证据转成可引用表述。做具体结论时，尽量带上模块名、类名、配置项或文件路径。

如果当前信息不足以支撑结论，停止继续摸底，改为下调断言强度，并用 `基于当前仓库能确认的范围` 或 `如果你的职责覆盖这里，可以这样回答` 这类表达继续组织答案。

### 2. 处理最小信息缺口

补信息阶段遵守这些规则：

- 仅在无法支撑当前输出时，补一个最高价值证据缺口
- 默认通过仓库入口、配置和代码定位来补，不先向用户发起信息采集
- 如果仍然缺少用户侧事实，就在答案里保留边界说明或条件句，而不是中断问答流程
- 信息一旦足够，立即回到第 6 节规定的默认输出

默认提问优先级见 `references/initialization-playbook.md`。

### 3. 设计题目

出题时先按”维度定义”章节确定当前技术岗位/级别，再结合 `references/interview-playbook.md` 选择题型、追问链路和评分维度。需要校准结构化面试方法、题型混排、评分维度或 STAR 重心时，再读取 `references/external-benchmarks.md`，但不要把外部资料当成当前项目事实来源。

围绕同一个主题，按三层深度出题：

1. 基础识别题：判断候选人是否真的做过该项目
2. 深挖追问题：逼出架构、边界、异常流和取舍
3. 压力题：追问故障、性能瓶颈、线上事故、复盘和反思

以下字段仅作出题内部参考，不作为默认输出格式；实际输出格式以第 6 节为准。

每道题在内部设计时都要想清楚：

- `出题意图`
- `题目正文`
- `优秀回答应覆盖的要点`
- `可继续追问的方向`
- `面试官评分关注点（参考 interview-playbook.md 第 4 节的 5 维度评分法）`

出题时优先采用结构化面试思路，而不是临场自由发挥。尽量把问题分配到固定类别：

- `验证题`：核验候选人是否真的接触过该系统、模块或场景
- `行为题`：要求候选人讲一个真实经历，观察其决策和推进方式
- `情景题`：给一个线上问题或架构权衡，观察其分析与应对
- `技能深挖题`：围绕链路、组件、代码实现和异常处理做技术深挖

在连续回合里，尽量让这四类问题都有覆盖。

题目正文必须符合真实面试口径：

- 使用面试官直接发问的口吻，不输出“出题意图”“考察点”这类元信息
- 问法要足够具体，能把候选人的真实经历、个人动作和边界处理逼出来
- 优先使用“你在这个项目里……”“你当时为什么……”“如果线上出现……你会怎么处理”这类句式
- 默认把题目当成现场首问，不在同一轮里把主问题和追问堆成多问连发
- 优先锚定当前仓库里证据最强的模块、链路、配置或异常场景，不问脱离项目上下文的通用八股
- 少用“介绍一下项目”这类过宽问题；即使是开场题，也要限制在职责、链路、结果或取舍中的一个明确焦点
- 问句优先控制在 `1~2` 句，口语化、直接、有压迫感，但不要写成审讯式长串追问
- 不在题目里暗示标准答案结构，不提前把“从背景、方案、结果说一下”这类答题框架塞给用户
- 对行为题，优先要求“具体讲一次”；对技术题，优先追问“为什么这么做”和“异常时怎么办”；对压力题，优先追问“先止血还是先定位，以及顺序为什么这样定”

优先覆盖以下主题：

- 项目介绍与业务闭环
- 微服务拆分与领域边界
- 核心调用链路
- 事务一致性与幂等
- 缓存、搜索、消息队列
- 高并发场景
- 安全、权限、风控
- 可观测性、排障、稳定性治理
- 团队协作、推进方式、取舍与复盘

详细题型和问题模板见 `references/interview-playbook.md`。

以上为选题参考框架；实际如何落到单轮输出，统一遵守第 6 节。

### 4. 构建答案

构建答案时直接按”维度定义”章节的技术岗位/级别调整重心。需要校准回答结构、个人动作占比、评分项解释或项目故事复用策略时，再读取 `references/external-benchmarks.md`。

用候选人口吻回答，但只基于仓库事实和合理推断；答案结构要显式服务于面试官的评分标准，而不是写成背诵稿。

优先采用这个结构：

`背景 -> 职责 -> 方案 -> 关键细节 -> 取舍 -> 结果 -> 复盘`

如果用户需要“单题标准回答”，优先退化成强化版 STAR：

- `Situation`：只交代必要背景，不堆细节
- `Task`：明确你负责什么、目标是什么
- `Action`：作为回答主体，重点写你亲自做了什么
- `Result`：写结果、指标、收益或至少写观察到的变化

控制讲述重心时，优先把篇幅压在 `Action`。如果答案明显在背景铺陈上过长，主动收缩背景，补充动作、权衡和结果。

写答案时遵守这些约束：

- 优先用第一人称，但不要虚构“我主导了全部模块”
- 没有直接证据时，明确写成“如果你的职责覆盖这里，可以这样回答”
- 能引用代码证据时，写清模块和实现位置
- 能量化再量化；不能量化就写机制和观察指标，不要编数字
- 解释“为什么这样做”，不要只罗列组件名
- 对中高级岗位，必须补充失败场景、边界条件和替代方案
- 默认输出成首轮口答版本，优先控制在 `45~90 秒` 可讲完的长度，除非用户明确要求展开
- 语言要像真实面试回答，句子短、信息密度高，避免写成长段报告或教程
- 不输出“这道题考察什么”“回答思路”“可继续追问”这类分析说明

输出前执行"真实性约束"章节的检查。

### 5. 贴合目标岗位

- 初始化阶段先记录默认技术岗位/级别推断，必要时允许后续回合修正
- 设计题目和构建答案时，统一按”维度定义”章节的技术岗位/级别选择重心
- 如果用户明确给出岗位或级别，覆盖默认推断

### 6. 唯一输出模板

所有输出格式规则集中在本节。每轮严格遵守以下结构：

```
## 题目
[面试官口吻的单个问题]

## 参考回答
[候选人第一人称答案，按背景→职责→方案→细节→取舍→结果→复盘组织，并满足真实性、个人贡献清晰度、技术深度、权衡意识、结果与复盘这五类面试官评估标准]
```

**必填**：`题目`、`参考回答`

**不允许出现**：项目卡片、题库列表、多道题、附加区、背诵版、口语版、模式说明、分析说明

默认行为：

1. 当前会话已存在 `/init` 结果或稳定项目基线 → 直接复用，不重复初始化
2. 初始化未完成 → 只补一个最关键仓库证据入口，不向用户追问前置信息
3. 初始化完成 → 直接进入第一题，不额外展示项目卡片或会话基线
4. 用户说”继续” → 优先选择四类题型（验证题/行为题/情景题/技能深挖题）中覆盖最少的一类，并把问题难度从“确认做过”逐步推到“边界、异常、取舍、复盘”
5. 用户说”继续”且当前链路证据仍够用 → 优先复用上一轮证据，不重新扫描无关模块；只有当新题明显跨链路时才补新的最小证据

## 真实性约束

始终遵守以下规则：

- 不虚构线上指标、团队规模、职责范围、事故经历
- 不把 README 里的宣传文案直接当成候选人的个人贡献
- 不把“项目支持某能力”直接说成“你亲自完成该能力”
- 不确定时直接标记为 `推断`，或改写成带边界的条件表达
- 发现 README、代码、模块数量存在不一致时，优先提示并解释口径差异

输出行为题答案时，优先使用 `我` 而不是 `我们`，除非必须解释团队配合。这样可以把个人动作和团队结果拆开，避免贡献模糊。

## 外部校准要点

结合外部高质量面试资料，默认加入这些校准：

- 行为题和项目题都尽量要求“具体例子”，不要接受纯观点回答
- 在连续回合里混合验证题、行为题、情景题和技能深挖题
- 回答是否优秀，不只看技术名词，还看是否讲清楚决策、动作、结果和复盘
- 候选人优先准备 3 到 5 个可复用项目故事，再映射到不同题目，而不是每题临时编故事
