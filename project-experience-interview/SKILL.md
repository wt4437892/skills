# Project Experience Interview

## 概览

先做轻量初始化，确认仓库范围、候选人职责、目标岗位和当前目标输出，再进入严格的逐轮一题一答。始终把内容分成三类：`已确认事实`、`合理推断`、`需要用户补充`，避免把通用八股硬套成项目经历。

## 核心交互规则

- 每轮默认只输出当前这一题和对应答案；如果初始化信息不足，只补一个最关键缺口。
- 不在同一轮里抛多个独立问题，不一次性给多道题，不生成整套题库。
- 如果用户只说“继续”，选择当前信息增益最高的下一题输出。
- 在初始化完成前，优先限制仓库探索范围，不直接深翻大批代码。
- 发现信息不足时，优先追问，不用长时间自行摸底替代用户补充。

## 工作流

### 0. 初始化项目上下文

先读 `references/project-context.md` 和 `references/initialization-playbook.md`，把当前会话初始化成一个可持续补全的项目卡片。

初始化阶段只允许读取这些高价值入口：

- `README.md`
- `pom.xml`
- `package.json`
- `AGENTS.md`
- 仓库根目录模块列表
- `references/project-context.md`
- `references/initialization-playbook.md`

初始化时优先确认：

- 项目一句话定位
- 目标岗位或使用场景
- 用户实际负责的模块、链路或专题
- 最想突出的亮点、难点或结果
- 当前最想先产出的内容

初始化未完成时，不继续深挖代码，先问一个最关键的问题。默认优先确认 `你实际负责哪个模块或哪段链路？`

初始化完成的最低条件：

- 已知项目定位
- 已知用户职责范围
- 已知目标岗位或当前用途
- 已知这轮要先产出什么

完成后，先用 3 到 6 行输出当前会话基线，再进入第一题输出，只保留关键信息，不展开长文。

### 1. 定点建立项目画像
初始化完成后，再围绕当前问题做定点验证。只读与当前问题直接相关的模块目录、配置文件、核心 Service、Controller、Mapper，不做全仓库扫描。

优先验证这些与当前问题直接相关的信息：

- 当前模块或链路的关键实现位置
- 当前题目对应的边界、异常流、状态流转和依赖关系
- 当前职责范围内可落到个人动作的难点、优化点和取舍点
- 支撑当前回答所需的配置项、类名、方法名、表名或消息主题

把仓库里的明确证据转成可引用表述。做具体结论时，尽量带上模块名、类名、配置项或文件路径。

如果当前信息不足以支撑结论，停止继续摸底，改为发起一个定点追问。

### 2. 补齐最小信息缺口

补信息阶段遵守这些规则：

- 仅在无法支撑当前一题一答时，补一个最高价值缺口
- 补充问题必须具体、可直接回答，避免并列小问
- 用户回答后，先回写 `已确认事实` 和 `仍待补充`
- 信息一旦足够，立即回到当前这一题的题目和答案输出

默认提问优先级见 `references/initialization-playbook.md`。

### 3. 设计题目

出题时直接按目标岗位调整方向选题：Java 后端优先服务拆分、RPC、事务、缓存、SQL、排障；架构师或高级工程师优先边界划分、演进路径、治理策略、容量评估；面试官视角优先区分度、追问点、打分标准、真假识别；候选人备面优先可信度、可讲述性、细节密度、前后一致。

围绕同一个主题，按三层深度出题：

1. 基础识别题：判断候选人是否真的做过该项目
2. 深挖追问题：逼出架构、边界、异常流和取舍
3. 压力题：追问故障、性能瓶颈、线上事故、复盘和反思

以下字段仅作出题内部参考，不作为默认输出格式；实际输出格式以步骤 6 的模式定义为准。

每道题都给出：

- `出题意图`
- `题目正文`
- `优秀回答应覆盖的要点`
- `可继续追问的方向`
- `面试官评分关注点（参考 interview-playbook.md 第 4 节的 5 维度评分法）`

出题时优先采用结构化面试思路，而不是临场自由发挥。尽量把问题分配到固定类别：

- `验证题`：核验候选人是否真的接触过该系统、模块或场景
- `行为题`：要求候选人讲一个真实经历，观察其决策和推进方式
- `情景题`：给一个线上问题或架构权衡，观察其分析与应对
- `技能深挖题`：围绕链路、组件、代码实现和异常处理做技术深挖

在连续回合里，尽量让这四类问题都有覆盖。

优先覆盖以下主题：

- 项目介绍与业务闭环
- 微服务拆分与领域边界
- 核心调用链路
- 事务一致性与幂等
- 缓存、搜索、消息队列
- 高并发场景
- 安全、权限、风控
- 可观测性、排障、稳定性治理
- 团队协作、推进方式、取舍与复盘

详细题型和问题模板见 `references/interview-playbook.md`。

以上为选题参考框架，每轮仍只从中选一道题输出。

### 4. 构建答案

构建答案时直接按目标岗位调整重心：Java 后端优先强调服务拆分、RPC、事务、缓存、SQL、排障；架构师或高级工程师优先强调边界划分、演进路径、治理策略、容量评估；面试官视角优先强调区分度、追问点、打分标准、真假识别；候选人备面优先强调可信度、可讲述性、细节密度、前后一致。

用候选人口吻回答，但只基于仓库事实和合理推断。

优先采用这个结构：

`背景 -> 职责 -> 方案 -> 关键细节 -> 取舍 -> 结果 -> 复盘`

如果用户需要“单题标准回答”，优先退化成强化版 STAR：

- `Situation`：只交代必要背景，不堆细节
- `Task`：明确你负责什么、目标是什么
- `Action`：作为回答主体，重点写你亲自做了什么
- `Result`：写结果、指标、收益或至少写观察到的变化

控制讲述重心时，优先把篇幅压在 `Action`。如果答案明显在背景铺陈上过长，主动收缩背景，补充动作、权衡和结果。

写答案时遵守这些约束：

- 优先用第一人称，但不要虚构“我主导了全部模块”
- 没有直接证据时，明确写成“如果你的职责覆盖这里，可以这样回答”
- 能引用代码证据时，写清模块和实现位置
- 能量化再量化；不能量化就写机制和观察指标，不要编数字
- 解释“为什么这样做”，不要只罗列组件名
- 对中高级岗位，必须补充失败场景、边界条件和替代方案

输出前执行"真实性约束"章节的检查。

如果当前为候选人模式，在标准答案基础上额外生成两个版本：
- 背诵版：压缩到 5 句以内，保留核心动作和结果
- 口语版：用自然对话语气改写，去掉术语堆砌

如果用户在做模拟面试，默认流程是：

1. 先输出一道题
2. 等用户作答
3. 再给点评、示范回答或追问
4. 用户说“继续”后再进入下一题

### 5. 贴合目标岗位

根据目标调整题目和答案重心：

- Java 后端：强调服务拆分、RPC、事务、缓存、SQL、排障
- 架构师/高级工程师：强调边界划分、演进路径、治理策略、容量评估
- 面试官视角：强调区分度、追问点、打分标准、真假识别
- 候选人备面：强调可信度、可讲述性、细节密度、前后一致

### 6. 输出固定格式

默认每轮只输出 `当前这一题 + 对应示范答案`。

如果用户没有指定格式，默认行为是：

1. 先判断初始化是否完成
2. 信息不足则只补一个最关键问题
3. 信息足够则输出当前信息增益最高的一题和对应答案
4. 等用户说"继续"后，优先选择四类题型（验证题/行为题/情景题/技能深挖题）中覆盖最少的一类出下一题。

如果用户明确要求“面试官模式”，当前这一题至少包含：

- `识别信号`：什么回答说明候选人真的做过
- `减分信号`：什么回答像背八股或虚构经历
- `建议追问`：下一问如何把回答压实

如果用户明确要求“候选人模式”，当前这一题至少包含：

- `可直接背诵版`
- `面试口语版`
- `待你补充的信息`

## 真实性约束

始终遵守以下规则：

- 不虚构线上指标、团队规模、职责范围、事故经历
- 不把 README 里的宣传文案直接当成候选人的个人贡献
- 不把“项目支持某能力”直接说成“你亲自完成该能力”
- 不确定时直接标记为 `推断` 或 `待补充`
- 发现 README、代码、模块数量存在不一致时，优先提示并解释口径差异

输出行为题答案时，优先使用 `我` 而不是 `我们`，除非必须解释团队配合。这样可以把个人动作和团队结果拆开，避免贡献模糊。

## 外部校准要点

结合外部高质量面试资料，默认加入这些校准：

- 行为题和项目题都尽量要求“具体例子”，不要接受纯观点回答
- 在连续回合里不要只问经历题，要混合验证题、行为题、情景题和技能深挖题
- 回答是否优秀，不只看技术名词，还看是否讲清楚决策、动作、结果和复盘
- 面试官输出尽量采用固定评分维度，减少主观“感觉不错”
- 候选人输出尽量准备 3 到 5 个可复用项目故事，再映射到不同题目，而不是每题临时编故事
