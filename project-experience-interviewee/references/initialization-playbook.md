# 会话初始化指引

## 目录

1. 目标
2. 最小初始化字段
3. 提问优先级
4. 单轮模板
5. 仓库阅读预算
6. 结束条件

## 1. 目标

初始化阶段的目标不是摸透整个仓库，而是快速建立一个足够驱动后续问答的项目卡片，避免一开始就在陌生项目上做大范围代码阅读。默认依赖仓库事实自行建上下文，不把用户职责、目标岗位级别、当前优先交付当成前置必填。若 `project-context.md` 仍是其他项目的示例内容，只把它当模板，不把其中的项目事实直接带入当前会话。

如果当前会话已经执行过 `/init`，或者前文已经稳定确认过项目定位、技术栈、核心模块和主要链路，就直接把这些内容当作初始化结果复用，不再重复走初始化摸底流程。

## 2. 最小初始化字段

至少补齐以下字段后，再进入深挖：

- `项目定位`
- `主技术栈和模块边界`
- `至少一个证据足够强的模块、链路或专题`
- `默认技术岗位/级别推断`
- `可持续补全的项目卡片`

如果用户主动补充了职责范围、目标岗位、亮点、难点、结果、指标，也记录进项目卡片，但这些不是初始化完成的硬门槛。

## 3. 提问优先级

默认先做仓库侧补全，不主动向用户发起初始化追问。按信息增益从高到低优先补这些内容：

1. `项目最核心的业务闭环和主技术栈是什么？`
2. `仓库里最容易落到代码证据的模块、链路或专题是什么？`
3. `哪一个实现点最适合转成高质量面试问答？`
4. `如果用户没有指定技术岗位/级别，默认应映射到什么方向？`（Java 后端 或 架构师/高级工程师）

选择问题时遵守两条规则：

- 优先问会改变后续仓库阅读范围的问题
- 优先问能直接决定当前输出形式的问题
- 一旦已经满足初始化最低条件，立即停止继续补证，不为了“信息完整”继续补读低价值入口

## 4. 单轮模板

### 初始化输出模板

内部先形成项目卡片，例如：

- `项目定位`
- `主技术栈`
- `候选模块/链路`
- `默认技术岗位/级别推断`
- `当前已确认事实 / 回答边界`

然后按 `SKILL.md` 第 6 节进入固定的一问一答输出。

如果必须向用户补充信息，只允许补一个会明显改变答案真实性的关键事实；否则一律不断流。

## 5. 仓库阅读预算

初始化完成前，只读这些入口：

- `README.md`
- `pom.xml`
- `package.json`
- `AGENTS.md`
- 仓库根目录模块列表
- `references/project-context.md`
- `references/initialization-playbook.md`

这些入口按需读取，不要求全部读完：

- 优先看最可能快速确定项目定位和模块边界的文件
- 优先读取文件的高价值片段、标题、依赖块、模块声明，不默认整文件通读
- 满足初始化结束条件后立即停止，不继续补读剩余入口

使用 `references/project-context.md` 前，先确认它是否已经替换为当前项目内容；若未替换，只借用其字段结构，不采纳其中的项目专有结论。

初始化完成后，才允许围绕当前问题定点阅读对应模块。如果仓库很大，也优先用模块名、配置、依赖关系和入口类缩小范围，而不是先向用户追问职责。

如果当前会话已经有 `/init` 结果，可直接把它当成阅读预算之外的上游上下文使用；只有当 `/init` 结果不足以支撑当前问题时，才回到本节的最小补证流程。

连续回合里，优先复用最近一轮已经确认过的证据清单。只有当题目切换到新的模块、链路或故障域时，才新增读取目标。

## 6. 结束条件

满足以下条件即可认为初始化完成：

- 能用一句话解释项目是做什么的
- 能定位至少一个证据充分的模块、链路或专题
- 能推断默认技术岗位/级别（Java 后端 或 架构师/高级工程师）
- 已形成可持续补全的项目卡片

达到结束条件后，后续输出节奏统一遵守 `SKILL.md` 第 6 节的一问一答规则；项目卡片只作为内部上下文，不对用户展示。
